Where the engine lives.

Today the engine runs natively on Mac, Windows, and Linux. Three builds. Three sandboxing models. Going forward, the engine always runs inside a Linux microVM.

The shift

When you install Houston on your Mac, the desktop app spins up a tiny Linux VM and runs the engine inside it. When you sign up for Houston Cloud, a tiny Linux VM is provisioned for you in the cloud and the same engine runs inside it.

The engine binary is byte-for-byte the same. The only difference is whose computer the VM runs on.

Why a VM. Four reasons.

  1. One binary. No more Mac, Windows, and Linux builds of the engine. Linux only.
  2. One sandbox model. Linux primitives. landlock, namespaces, per-user permissions. No more sandbox-exec on Mac and AppContainer on Windows.
  3. Identical to production. What runs on your laptop is bit-identical to what runs in our cloud. Zero "works on my machine".
  4. Strong isolation. A VM is a kernel-level wall between Houston and the rest of your computer. Your kid can't accidentally delete the wrong file. An accidentally-installed agent can't read your bank statements.

The cost

About 2 seconds of cold start when you open the app, and roughly 100MB of extra memory. We think it's worth it for the simplicity and the security story.

Files live inside the VM

The VM has its own disk volume. All agent files, project files, everything lives inside the VM. The host doesn't see them. No virtio-fs file sharing, no performance penalty.

To import a folder, you drop it in via the app. It copies in once. To export, you download via the app. To back up, you snapshot the volume.

The big simplification

Desktop and cloud become the same architecture. The only difference is whose computer the VM runs on.

VM hypervisors per host OS

macOS: Apple Virtualization.framework. Windows: Hyper-V or Windows Hypervisor Platform. Linux: KVM via libvirt. Each is wrapped behind a small platform-specific helper in the desktop shell.
