The roadmap.

Three milestones. Each one roughly two to four weeks at current shipping pace. Each one ships before the next one starts.

M1 — Reliability

Goal: conversations never drop bytes. The "my message disappeared" failure mode goes away. Every byte the user sees is journaled before broadcast.

  • Add four SQLite tables: turns, turn_stream, events_outbox, cron_runs.
  • Write rows BEFORE doing work. Sweep orphans on boot.
  • WebSocket replay by sequence number on reconnect.
  • Cron audit log plus missed-fire catch-up policy.

Mostly inserts and middleware. No architectural change. About 2-3 weeks.

M2 — Auth and isolation

Goal: multi-tenant safety. Two humans can share an engine without leaking into each other. Per-agent credentials.

  • Typed tokens with scope: (principal_id, scope) on every route.
  • Per-agent Linux users inside the VM (one-time automated useradd).
  • Per-agent credentials (per-agent .claude, .composio).
  • VM-based desktop spin-up, replacing the native sidecar model.

Middleware plus a new system layer. About 3-4 weeks.

M3 — Cloud and detached workers

Goal: Houston Cloud is shippable. Engine bounces don't kill conversations. Sleeping engines wake on demand.

  • Detached houston-turn-worker binary so engine restart doesn't kill in-flight turns.
  • POST /v1/hooks/<route_token> wired to the event queue.
  • Engine knows how to exit gracefully on idle.
  • Relay knows how to wake a dormant engine via Fly Machines API.
  • Cron moves to a central scheduler service for hosted mode.
  • Per-customer Fly Machine provisioning and onboarding.

The only milestone with real architectural risk (the detached worker). About 4 weeks.

Open questions to settle before each milestone

Before M2: per-agent Claude logins, or a credentials broker for one-login-many-agents?

Before M3: cron in-engine forever, or move to a central scheduler even for self-hosters? Where does the OSS-vs-cloud line sit for the relay code?

After M3

Houston Cloud private beta is shippable. The desktop story is the same architecture as the cloud story, just running on the user's hardware. Pricing aligns (per VM-hour or per agent-month, decide then). Teams becomes a thin layer on top of the per-customer-VM story.
