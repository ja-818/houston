Why conversations die today.

The engine writes to disk only after the work is done. Everything in flight is in-memory. In-memory means it dies when the process dies. Here are the four failure modes.

Failure mode #1

Engine crashes between "accept message" and "spawn CLI". Your message disappears. No record anywhere. You click send again, the engine has no memory of the first attempt.

Failure mode #2

Engine crashes mid-stream. Whatever the assistant had already streamed: gone. Database has nothing. The user thinks the agent is broken.

Failure mode #3

WebSocket reconnects after a network blip. Server has no record of what events it pushed during the disconnect. The frontend has to refetch state and pray nothing slipped through.

Failure mode #4

Cron fire happens while engine is restarting. The 7am alarm rings, the engine is rebooting, the fire is dropped silently. No log, no alert, no catch-up.

The pattern

Notice the common thread. The engine writes to the database after doing work, not before. So anything happening between "before" and "after" lives only in memory, and that memory belongs to a process that can die.

The fix is boring but life-changing. Flip the order. Chapter 6 shows how, with the four SQLite tables that make it work.

Why this hasn't been fixed yet

Because it works most of the time. Engines don't usually crash. Networks don't usually blip. So the team shipped features instead. The point of the redesign is to make Houston work reliably for the cases where things DO go wrong.
