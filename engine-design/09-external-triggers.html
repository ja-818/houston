Chapter 09 Partial

External triggers.

Three ways something outside Houston can wake up an agent. Two of them work today. One is half-built.

1. The user sends a message

Standard chat. The desktop UI calls POST /v1/agents/<path>/sessions. The engine spawns a CLI subprocess, the response streams back. This is what Chapter 4 covers. Works today.

2. A cron fires

Each agent has routines. 0 7 * * * runs at 7am every day. Today the cron lives inside the engine. A tokio task sleeps until the fire time, then pushes a HoustonInput::Cron onto the event queue, which dispatches to the same chat-turn path.

This works as long as the engine is up. For sleeping cloud engines (Chapter 10), the cron will move to a central scheduler that wakes the engine when the time comes.

3. A webhook arrives

Stripe sends a payment event. Slack sends a message. GitHub sends a PR notification. Each one routed to a specific agent.

The plumbing exists. The engine has a typed HoustonInput::Webhook enum, and the dispatcher knows how to route it. What's missing is the HTTP route that constructs one. Adding POST /v1/hooks/<route_token> is small work that unlocks a huge surface.

Per-hook tokens, not the bearer token

Each webhook gets its own route token, separate from the engine bearer. Stripe's token only fires the Stripe hook. A compromised Stripe token lets the attacker fake payment events but doesn't let them read your chat history.

Wake-on-event for sleeping engines

Once the engine learns to sleep (Chapter 10), the relay in front will catch inbound triggers, wake the engine via the platform API (Fly Machines, systemd socket activation, etc), wait for /v1/health to respond, then proxy the request through.

Cron, user message, and webhook all share that same wake-up path. One mechanism, three triggers.

Where to look in the code

Webhook input type: engine/houston-events/src/queue.rs:25-104. Cron implementation: engine/houston-scheduler/src/cron_job.rs. Missing piece for webhooks: a routes/hooks.rs file in engine/houston-engine-server/.
