Deployment: one binary, two homes.

Same engine binary. Two deployment modes. Almost zero divergence between them. The only difference is whose computer the VM runs on.

The engine binary is byte-for-byte the same. Only the host changes.

Desktop mode

Inside a Linux microVM on your laptop. The microVM lives next to your desktop shell. They talk over local networking. The engine never crosses the VM boundary unless the shell explicitly bridges something (file picker, OAuth callback, system browser).

Cloud mode

Inside a Linux microVM on Fly Machines (or similar). Each paying customer gets their own VM. A relay outside knows how to wake a sleeping VM when a user message, webhook, or scheduled cron arrives.

The relay is the same one Houston already uses for mobile traffic (houston-relay/, a Cloudflare Worker plus Durable Objects on tunnel.gethouston.ai). Extending it to also proxy general engine traffic and to dispatch wake-ups is the bulk of M3.

What you get

  • One engine codebase, no forks, no per-platform builds beyond the small desktop shell.
  • Same tests, same behavior, same bugs and same fixes.
  • "Run your engine in our cloud" is literally fly machines start.
  • "Move from our cloud to your own VPS" is docker pull && docker run. Same image.
  • Pricing aligns if you bill per VM-hour: cloud customers pay you, desktop users pay their own machine, you meter the same unit.

Sleeping engines

For cloud mode, engines sleep after N idle minutes. They exit 0 gracefully. The relay holds the connection from the next inbound request, calls the platform's start API, polls /v1/health, then proxies through. Cold start: ~2 seconds. Acceptable for crons and webhooks; for active human chats, engines stay sticky-hot during a session window.

The killer property

You can no longer have a "works on my Mac" bug, because there is no Mac engine. Every engine is the same Linux engine.

What's there already

Always-on Docker recipe: always-on/Dockerfile and docker-compose.yml. Relay: houston-relay/. Tunnel client crate: engine/houston-tunnel. What's missing: VM-management code in the desktop shell, wake API in the relay, idle-exit logic in the engine.
