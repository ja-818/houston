Chapter 06 Planned · M1

The four tables that save everything.

Four new SQLite tables. That's the entire reliability story. Not Temporal, not a workflow engine. Just four tables and one rule: write the row before doing the work.

The tables

TableWhat it holdsWhat it fixes
turns Every chat turn, with status Crash before CLI spawns. Retry is idempotent via turn_id.
turn_stream Every assistant token chunk, with seq Reconnect with since_seq, perfect replay.
events_outbox Every event with a monotonic seq WS clients reconnect and get the gap they missed.
cron_runs Every cron fire (or missed fire) Missed at 7am, caught up on next boot.

The rule

Write the row before doing the work. Mark it complete after. Here is what that looks like for a single chat turn. 

1. Accept message       → INSERT turns (status: queued)
2. Start CLI            → UPDATE turns SET status = running
3. Stream chunk         → INSERT turn_stream row, then broadcast
4. CLI done             → UPDATE turns SET status = completed
5. Engine boots         → SELECT * FROM turns WHERE status = running
                          → mark orphaned, surface retry

Step 5 is the magic. On every boot, the engine looks for turns that were "running" when the previous engine died. Those are orphans. The user sees a retry button. They click it. The same turn_id goes back into the system. We never accidentally cook the same dish twice because the engine recognizes the ID and resumes.

Why this isn't Temporal

Temporal is a workflow engine for orchestrating long-running multi-step business processes with replay semantics. Houston has one activity: run a CLI turn, capture the stream. That's a job, not a saga. Four tables are enough.

The one architectural piece

Everything above is just "add a table". One piece is genuinely harder. Today the CLI subprocess is a child of the engine process, so killing the engine kills the CLI. To survive engine restart mid-stream (not just on the next turn), we need a small houston-turn-worker binary.

The engine spawns it detached. The worker writes streaming output to SQLite directly. The engine bounces, a new engine starts up, reattaches via Unix socket, conversation never blinks.

This piece is M3. Until then, the cheap fallback (orphan + retry) gets you 70% of the user experience for 10% of the work.

Migration order

SQLite migrations live in engine/houston-db/src/migrations.rs. New tables drop in next to existing chat_feed, preferences, and engine_tokens. Each gets a numbered migration file; existing data is preserved.
