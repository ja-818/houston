Chapter 07 Planned · M2

Keeping agents apart.

Three independent walls between agents. Each one is enough to stop a curious agent from snooping. Together, they make accidental and adversarial cross-agent reads impossible.

Sales agent

  • Linux user: hou_sales
  • Folder: /agents/sales/ (mode 0700)
  • Claude login: /home/hou_sales/.claude/
  • Composio: /home/hou_sales/.composio/
  • Tools: Slack, CRM, Email
NO ACCESS

HR agent

  • Linux user: hou_hr
  • Folder: /agents/hr/ (mode 0700)
  • Claude login: /home/hou_hr/.claude/
  • Composio: /home/hou_hr/.composio/
  • Tools: Payroll, Benefits, Workday

Wall 1: per-agent Linux user

Each agent gets its own Linux user inside the VM. Their folders are owned by them, mode 0700. When you send a message to the Sales agent, the CLI runs as hou_sales. It physically cannot read hou_hr's files. The kernel says no.

Even if a clever prompt asks the Sales agent to read HR salary data, the agent has no permission. There is no jailbreak that gets around chmod.

Wall 2: per-agent credentials

Each agent has its own ~/.claude, its own ~/.composio, its own everything. When Sales logs into Anthropic, the token sits in /home/hou_sales/.claude/. HR can't see it.

Even if Sales gets compromised, its blast radius is bounded by what it had access to. It can't social-engineer its way to HR data because it doesn't have the keys to start with.

Wall 3: engine-level scope checks

On top of the file walls, every HTTP route checks "does this user's token have scope for this agent?" before doing anything. A user with Sales access trying to query HR's data gets a 403 before the engine even reaches for the filesystem.

Tokens become typed: (token_id, principal_id, scope: ["WorkspaceX/Sales/**"]). Middleware enforces. New tables: principals and principal_tokens in houston-db.

The headline

"My HR agent literally cannot leak salaries to my Sales agent, even if jailbroken." That isn't marketing copy. That's what the kernel enforces.

Where this matters

This is a real positioning win for regulated industries. ChatGPT Enterprise and Claude Teams put everyone in one tenant with one set of permissions. Houston's per-agent isolation is something they don't offer.

What changes in code

New auth middleware in engine/houston-engine-server. New tables in engine/houston-db. CLI spawn in engine/houston-terminal-manager gains a setuid step (Linux only; gated by HOUSTON_ISOLATE_BY_UID=1). Per-agent home dir resolution in houston-agent-files.
